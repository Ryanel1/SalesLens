## Anti-Patterns Verdict

**LLM assessment**: SalesLens does not look like a generic AI SaaS dashboard. The "operator scoreboard" direction is visible in the scorecards, product rankings, merchandise imagery, and account-specific accents. The failure mode is product-craft slop: too many similarly weighted cards, repeated uppercase micro-labels, dropdown-heavy controls, and a CSS history that can revive deprecated visual patterns.

**Deterministic scan**: The detector found 9 warnings in `saleslens-web/src`: 7 `side-tab`, 1 `overused-font`, and 1 `layout-transition`. The Inter warning is low-signal because Inter is explicitly documented as the product UI font. The side-tab findings are real design debt because `DESIGN.md` already says decorative side-stripe borders should be redesigned into badges, labels, or full-state treatments when touched.

**Visual overlays**: No reliable user-visible overlay is available. Browser mutation preflight failed because the in-app browser evaluate surface was read-only, so `detect.js` was not injected. Live browser evidence reached the unauthenticated login page, which rendered cleanly at desktop and mobile widths.

## Overall Impression

SalesLens has the bones of a distinctive private retail analytics tool. The dashboard knows its world: POS periods, YTD comparisons, inventory position, and product imagery. The biggest opportunity is to make the interface more opinionated about what Ryan should act on next. Right now it often presents a correct report wall instead of a decision sequence.

## What's Working

- The app has a real product point of view. Product imagery, rank badges, scorecard type, and inventory gauges carry useful personality without needing decorative fluff.
- The main information architecture is sensible: account and period controls lead into scorecards, monthly/weekly performance, inventory, and top performers.
- The current flat dashboard override layer is moving in the right direction: restrained surfaces, dark top navigation, compact controls, and account accents used mostly as data color.

## Priority Issues

**[P1] The dashboard does not consistently force the next decision**

**Why it matters**: SalesLens exists to make complex POS data fast to act on. The current sections show many useful metrics, but the hierarchy does not always answer "what changed, why, and what deserves attention?"

**Fix**: Give each major section a compact decision lead: change, driver, action/risk. Reduce equal-weight metric tiles where they repeat the same visual weight, and make the strongest operational takeaway the first thing in each section.

**Suggested command**: `$impeccable distill`

**[P1] Import and upload management are under-designed for high-risk data changes**

**Why it matters**: Uploading and deleting POS data changes the analytical truth of the app. The import modal asks users to choose file type after selecting files, but it does not preview detected type, date coverage, affected account, replacement impact, skipped rows, or record counts before committing. Delete uses `window.confirm` in [page.tsx](/Users/ryan/Documents/Codex/2026-05-19/need-to-build-mac-application-to/saleslens-web/src/app/page.tsx:1276), which feels too crude for a consequential operation.

**Fix**: Treat import as an approval flow: files -> detected type/range -> replacement impact -> import. Replace native delete confirmation with a structured destructive confirmation showing account, date range, rows, units, sales, and inventory records affected.

**Suggested command**: `$impeccable harden`

**[P1] Accessibility and keyboard confidence are not strong enough for custom controls**

**Why it matters**: The app uses custom dropdowns and several modal dialogs ([page.tsx](/Users/ryan/Documents/Codex/2026-05-19/need-to-build-mac-application-to/saleslens-web/src/app/page.tsx:1756), [page.tsx](/Users/ryan/Documents/Codex/2026-05-19/need-to-build-mac-application-to/saleslens-web/src/app/page.tsx:2035)). Source review found dialog semantics, but no broad Escape handling, focus trap, focus return, or global `:focus-visible` system beyond login inputs.

**Fix**: Add a consistent focus-visible token, Escape-to-close for modals/dropdowns, focus return after modal close, keyboard navigation for custom dropdown options, and non-color text summaries for charts/gauges.

**Suggested command**: `$impeccable audit`

**[P2] The visual system still contains mixed-era patterns**

**Why it matters**: `globals.css` has older atmospheric photo/glass/shadow styling followed by a later flat-dashboard override layer. Even when the cascade currently resolves correctly, future edits can accidentally reintroduce deprecated ghost-card, side-stripe, and glassy panel treatments.

**Fix**: Extract the current flat layer into clean tokens/components and retire or quarantine the older atmospheric layer to login/share-only styles. Rewrite side-stripe state treatments into badges, icons, labels, or full-row state.

**Suggested command**: `$impeccable extract`

**[P2] Product gallery controls compete with product browsing**

**Why it matters**: Top Performers should be the fun, fast reward of the app. The current header can expose View, Sort, Filter, Show, cache-image action/status, and dozens of cards, which asks the user to manage the report before enjoying the answer.

**Fix**: Keep the default gallery opinionated. Leave View and Sort visible, move advanced filters/show-count into one "Refine" control, and show active filter chips plus a single reset above results.

**Suggested command**: `$impeccable layout`

## Persona Red Flags

**Ryan, Operator/Analyst**: Ryan can see YTD, monthly, weekly, inventory, and products, but still has to decide which variance deserves action. Missing "attention needed" summaries slow repeated analysis.

**Alex, Power User**: No visible keyboard shortcuts, saved report presets, fast account switching command, or batch product/report actions. Upload management supports editing and deleting rows, but the workflow remains click-heavy.

**Sam, Accessibility-Dependent User**: Custom dropdowns and modals need stronger keyboard/focus guarantees. Focus styling is not broadly defined. Charts and inventory state rely partly on color/position with limited equivalent summaries.

**Riley, Stress Tester**: Import/delete/share flows rely on status strings and occasional generic errors. Refreshing or closing during long import/image-cache tasks could leave uncertainty about what completed.

## Minor Observations

- The top nav compact behavior hides the nav on scroll with opacity and pointer-events, which can feel surprising because filters are core controls.
- The login screen renders cleanly in desktop and mobile viewport checks, but it is visually more atmospheric than the flat dashboard system. That is acceptable if it stays quarantined.
- "Arts" is efficient for Ryan, but shared-report viewers may need "Artwork codes" or a tooltip.
- The public share report inherits the dashboard sections, but recipient context/help is thin compared with internal operator context.
- The detector's `layout-transition` hit points to a nav transition on layout properties. Product UI motion should convey state without animating layout where possible.

## Questions to Consider

- What would the dashboard look like if every section had to answer one sentence: "Pay attention to ___ because ___"?
- Should import be treated less like "upload a file" and more like "approve a data change"?
- Which four controls does Ryan actually use 90% of the time, and why are the others equally visible?
- Could Top Performers become the emotional reward of the product instead of another configurable report block?
