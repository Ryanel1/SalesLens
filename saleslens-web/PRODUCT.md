# Product

## Register

product

## Users

SalesLens is for Ryan, working as the operator and analyst of a private POS reporting tool. The main use context is reviewing customer sales and merchandise performance from imported POS data, quickly understanding what changed, and deciding where attention should go next.

## Product Purpose

SalesLens turns monthly POS, sales, product, and inventory data into a clear web dashboard for retail account analysis. It helps answer questions about sales trends, year-over-year performance, inventory risk, margin and retail value signals, top products, store or account performance, category performance, customer behavior, upload history, and shareable report snapshots.

The active product is the web app only. The older macOS app is historical context and should not drive future design decisions except where it explains the original data workflow.

Success means the dashboard makes complex POS data feel quick to scan, fun to view, and easy to act on without burying the user in controls or competing panels.

## Brand Personality

Fun to view, minimal, efficient.

SalesLens should feel like a capable private operator dashboard: confident, focused, responsive, and slightly lively without becoming decorative. It should reward repeated use with clear hierarchy, crisp comparisons, and moments of visual interest that help the data make sense.

## Anti-references

Avoid complexity for its own sake: overloaded dashboards, dense control clusters, repeated card grids, and panels that all compete for attention.

Avoid generic SaaS dashboard tropes, decorative gradients, bloated hero metrics, and visual flourishes that make POS analysis slower. Avoid treating SalesLens like a public marketing site; it is a working tool first.

Avoid letting the retired macOS app define the redesign. Preserve the useful data model and workflow knowledge, but design forward from the active web version.

## Design Principles

1. Start with the decision. Each screen should make the next operational question obvious: what changed, why it changed, and what deserves action.
2. Make dense data feel light. Use hierarchy, grouping, progressive disclosure, and focused defaults so the dashboard can cover many POS questions without feeling crowded.
3. Keep controls close to consequences. Filters, upload actions, share actions, and report navigation should sit where their impact is visible and understandable.
4. Let product imagery and merchandise identity carry delight. Product cards, top sellers, and inventory views can be fun to browse while staying compact and useful.
5. Design for repeat analysis. Favor fast scanning, stable layouts, reliable comparisons, and minimal friction over one-off visual drama.

## Accessibility & Inclusion

No special accessibility requirements were specified. Default to WCAG AA contrast, keyboard-accessible controls, visible focus states, color-blind-safe chart distinctions, responsive layouts, and reduced-motion alternatives for any animation.
